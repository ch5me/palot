## Bug fix this session
Problem: Hono route '/:laneId/*' with c.req.param('*') returned empty for sub-paths.
Effect:  /browser/default/assets/* was proxied as '/', returning the shell HTML
         (762 bytes) instead of the real JS/CSS bundles.
Fix:     Use c.req.path directly and strip the /browser/<laneId> prefix.
File:    apps/server/src/routes/browser-lanes.ts (lines 342-351)
Evidence before fix: 'JS asset: HTTP 200, 762 bytes' (shell HTML)
Evidence after fix:  'JS asset: HTTP 200, 694089 bytes' (real Selkies JS)

## Known gaps
1. CDP probe unreachable (/json/version). LinuxServer/Selkies Chromium ignores
   CHROME_CLI for --remote-debugging-port. Workaround: non-Selkies image or
   custom-cont-init.d writing chromium-flags.conf. Already documented in
   .sisyphus/evidence/browser-lanes/task-15-cdp.txt.

2. WebSocket bridge in Hono fetch proxy doesn't hold the upgraded connection
   open bidirectionally. fetch() returns the 101 + initial bytes from upstream
   as a Response body, but doesn't pipe further messages. Selkies server sees
   no keepalive pings and closes the upstream WS after ~10s:
     'Data WS closed with error from 127.0.0.1:44546: sent 1011 (internal
      error) keepalive ping timeout; no close frame received'
   Net effect: the page loads (Selkies shell + UI bundles) but the desktop
   stream does not appear, and input forwarding does not work.
   Workaround scope: replace fetch() proxy with a Bun-native upgrade handler
   (or proxy via a separate WebSocket server). Non-trivial; would need a new
   bridge between Hono and Bun.serve websocket handlers.
