# Task 9 — Zod-backed OpenCode Tool Projection <!-- oc:id=sec_aa -->

> Wave 2, Task 9 of plan `firefly-plugin-system-v2`. Do not modify the plan file.

## What this means for V2 <!-- oc:id=sec_ab -->

Every plugin surface produces paired, session-scoped, Zod-validated tool surfaces. The agent can list, inspect, open, and control plugin state without hand-written ad hoc bridge glue. Tool ownership is split between host-generated introspection/wrappers and plugin-implemented business tools, with one canonical result envelope and one 9-state tool-call state machine.

## 1. Tool ownership table <!-- oc:id=sec_ac -->

| Tool class | Declared in manifest | Implemented by plugin | Generated by host | Session-scoped | Notes |
|---|---|---|---|---|---|
| Introspection (`plugins.list`, `plugins.describe`, `plugins.tools`, `plugins.panels`, `plugins.widgets`, `plugins.commands`, `plugins.themes`, `plugins.state`, `plugins.permissions`, `plugins.lifecycle`) | no | no | yes | yes (mostly read-only) | Host derives from `PluginDescriptor` + `PluginInstance` |
| Surface inventory (`plugin.<id>.panel.list`, `plugin.<id>.widget.list`, `plugin.<id>.command.list`, `plugin.<id>.theme.list`) | optional metadata only | no | yes | yes | Host projection over contributions |
| Surface control wrappers (`plugin.<id>.panel.open`, `plugin.<id>.widget.mount`, `plugin.<id>.widget.unmount`, `plugin.<id>.command.run`, `plugin.<id>.theme.apply`, `plugin.<id>.theme.reset`, `plugin.<id>.theme.preview`) | yes | mixed | mixed | yes for panel/widget/command; app for theme | Host validates scope + capability, may call plugin handler or host adapter |
| Plugin business tools (`plugin.<id>.<verb>` from `contributes.tools`) | yes | yes | no | default yes | Plugin provides Zod schema and handler |
| Dynamic session tools | superset declared statically | yes | host may gate availability | yes | Runtime may hide/disable per session; declared names stay in the namespace |

## 2. Introspection tools (host-generated) <!-- oc:id=sec_ad -->

All host-generated, read-only or state-introspection. Read from `PluginDescriptor` + `PluginInstance` + `PluginSessionHandle`.

- `plugins.list` — every installed plugin, trust tier, lifecycle status
- `plugins.describe` — per plugin: id, version, capabilities, contribution count, instance status
- `plugins.tools` — every exposed tool across plugins with declared names, schemas, current session availability
- `plugins.panels` — every contributed panel with location, surface, capability needs
- `plugins.widgets` — every contributed widget with zone, default-enabled
- `plugins.commands` — every contributed command with shortcut, menu placements
- `plugins.themes` — every contributed theme with precedence, uiTheme class
- `plugins.state` — for a given plugin: instance health, crash counters, quarantine, last activation error
- `plugins.permissions` — for a given plugin: granted capabilities, granted network domains, default-deny list
- `plugins.lifecycle` — for a given plugin: lifecycle state, install source, available transitions

## 3. Surface control wrappers (host-generated, possibly plugin-implemented) <!-- oc:id=sec_ae -->

For each plugin contribution family, host exposes:

- panels: `plugin.<id>.panel.list`, `plugin.<id>.panel.state`, `plugin.<id>.panel.open`
- widgets: `plugin.<id>.widget.list`, `plugin.<id>.widget.state`, `plugin.<id>.widget.mount`, `plugin.<id>.widget.unmount`
- commands: `plugin.<id>.command.list`, `plugin.<id>.command.run`
- themes: `plugin.<id>.theme.list`, `plugin.<id>.theme.apply`, `plugin.<id>.theme.reset`, `plugin.<id>.theme.preview`

Implementation: wrapper validates Zod envelope, checks session scope and capability grant, then either calls plugin handler (if `handlerPluginOnly: true`) or calls host adapter (e.g. host applys theme). Wrapper returns canonical envelope.

## 4. Plugin business tools <!-- oc:id=sec_af -->

Declared in `contributes.tools[]`. The plugin provides:
- `name`
- `description` (model-readable)
- `inputSchema` (JSON Schema)
- `sessionScope`
- `timeoutMs`, `dispatchingCeilingMs`
- `requiresCapabilities[]`
- `uiHints` (optional)

Host validates input against the declared `inputSchema` via Zod, dispatches to the plugin's worker, and emits the standard envelope. Plugins cannot add tools at runtime outside their declared superset; the superset is fixed at install time.

## 5. Canonical result envelope <!-- oc:id=sec_ag -->

```ts
export type ToolEnvelope =
  | {
      status: "completed"
      data: unknown
      provenance: { pluginId: string; toolId: string; scope: ToolScope; capabilitiesUsed: string[] }
      uiHints?: Record<string, unknown>
      retryable?: false
    }
  | {
      status: "failed"
      errorCode: string
      errorMessage: string
      data?: unknown
      provenance: ToolEnvelopeBaseProvenance
      retryable?: boolean
    }
  | {
      status: "denied"
      errorCode: "permission_denied"
      errorMessage: string
      requiredCapability: string
      grantedCapabilities: string[]
      provenance: ToolEnvelopeBaseProvenance
    }
  | {
      status: "unavailable"
      errorCode: "tool_unavailable"
      errorMessage: string
      provenance: ToolEnvelopeBaseProvenance
    }
  | {
      status: "queued"
      queuePosition: number
      provenance: ToolEnvelopeBaseProvenance
    }
  | {
      status: "cancelled"
      errorCode: "tool_cancelled"
      errorMessage: string
      cancellationSource: "user" | "host" | "upstream"
      provenance: ToolEnvelopeBaseProvenance
    }
```

Error code taxonomy (canonical, host-reserved):

- `validation_error` — Zod parse failure
- `permission_denied` — capability broker rejected
- `tool_unavailable` — plugin disabled at dispatch time
- `tool_timeout` — host timer fired
- `tool_cancelled` — host or agent cancelled
- `bridge_unsupported_server` — server-mode matrix row (managed-only on attached)
- `no_active_server` — server-mode matrix row
- `session_lost` — server-mode matrix row
- plus plugin-emitted `errorCode` strings scoped to the plugin id

## 6. Tool-call state machine (9 states, integrated from plan) <!-- oc:id=sec_ah -->

```
queued ──► dispatching ──► running ──► completed
   │           │            │
   │           │            ├──► failed
   │           │            ├──► cancelled
   │           │            └──► timeout ──► failed
   │           │
   │           ├──► denied
   │           ├──► unavailable
   │           └──► cancelled
   │
   └──► cancelled
```

State semantics:

- `queued` — host accepted, awaiting worker slot. Envelope carries `queuePosition`.
- `dispatching` — host hands call to worker. Envelope carries `startedAt`. Ceiling 5s default.
- `running` — worker executing. Envelope carries `progress` updates. Ceiling 60s default.
- `timeout` — host timer fired. Always transitions to `failed` with `errorCode: tool_timeout`, `retryable: true`.
- `completed` — terminal success. Envelope carries typed `data`.
- `failed` — terminal. Envelope carries `errorCode`, `errorMessage`, `retryable`.
- `denied` — terminal. Capability broker rejected before dispatch. Envelope carries `requiredCapability`, `grantedCapabilities`.
- `unavailable` — terminal. Plugin worker offline or plugin disabled at dispatch.
- `cancelled` — terminal. Cancellable from any non-terminal state.

Default timeouts overridable per plugin in `contributes.tools[]` (`timeoutMs`, `dispatchingCeilingMs`).

## 7. Naming, collision, and reserved namespaces <!-- oc:id=sec_ai -->

- host reserves: `plugins.*`, `plugin.*` (with plugin-namespaced children), `firefly.*`, `surface.*`, `session.*`, `theme.*`, `sidebar.*`, `sidepanel.*`, `bridge.*`, `lifecycle.*`
- third-party plugins use: `plugin.<pluginId>.<verb>` for both their own tools and per-family wrappers
- collision detection: host rejects activation if the manifest declares any tool id that conflicts with reserved namespaces, or with another active plugin's tool id
- dynamic tools are NOT supported as new ids at runtime; only visibility of statically declared ids can change per session
