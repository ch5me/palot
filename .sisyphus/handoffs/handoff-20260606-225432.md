## High-Level Goal <!-- oc:id=sec_ab -->
Build a polished shared AI-composer input box for Elf that replaces the plain bordered prompt input with a premium animated shell. The target behavior is:
- crisp animated conic-gradient border using Elf brand colors
- WebGL glitter/sparkle border effect color-matched to the same conic gradient
- focus-driven state machine: border/glow/sparkles only appear when focused
- reusable shared implementation in `packages/ui` so all prompt surfaces inherit the same effect without duplicating logic

## Attempted Solutions <!-- oc:id=sec_ag -->
### 1. First shared composer shell pass <!-- oc:id=sec_ah -->
What was attempted:
- Introduced `InputGroupComposer` with gradient/glow styling and WebGL sparkle effect.
- Added `PROMPT_INPUT_COMPOSER_CLASSNAMES` in `prompt-input.tsx`.
- Migrated prompt consumers.

What worked:

What did not work:

- glow/sparkle sizing initially spilled beyond the input box

### 2. Oversized sparkle / missing conic ring debugging <!-- oc:id=sec_ai -->
Observed issue:
- sparkle was visible
- gradient border was missing or not visibly painted
- shell sometimes appeared larger than the input box due to overlay geometry

What was tried:
- clipped the composer shell bounds with overflow changes
- changed sparkle overlay bounds from oversized negative insets to fit the box
- adjusted border/glow class tokens

Root finding:
- a key bug was that the border layer class token existed but was not actually being passed into `InputGroupComposer` from `PromptInput`
- result: sparkle rendered because shader layer existed independently, but border layer had no real brand-gradient styling applied from the prompt-input side

Fix that was applied:
- `borderClassName={PROMPT_INPUT_COMPOSER_CLASSNAMES.border}` was wired into the `InputGroupComposer` call in `packages/ui/src/components/ai-elements/prompt-input.tsx`

### 3. Palette mismatch debugging <!-- oc:id=sec_aj -->
Observed issue:
- border/glow still looked like a generic light-blue gradient instead of Elf brand colors

What was tried:
- updated `inputGroupComposerPalettes` in `packages/ui/src/components/input-group.tsx`
- changed stops to use Elf lane colors (orange / amber / gold / violet)

Result:
- palette source now uses brand-colored stops rather than the initial blue-heavy fallback set

### 4. Focus-only state behavior update <!-- oc:id=sec_ak -->
Observed issue:
- user wanted the effect visible only on focus
- old focus blur/ring from the underlying inputs still competed visually with the composer treatment

What was tried:
- changed composer border token to start at opacity 0 and reveal on `group-focus-within/input-group-composer`
- changed composer glow token to start hidden and reveal on focus
- introduced internal focus tracking in sparkle overlay to show sparkles only while focused
- removed/reduced inherited focus ring treatment in shared `InputGroup`

Current caveat:
- this state-machine refactor happened while the border wiring was still being debugged, so a new agent should re-check the final live behavior end-to-end rather than assume it is visually perfect

### Important gotchas <!-- oc:id=sec_au -->
1. The WebGL sparkle layer can work even when the border layer is broken. Do not assume sparkle == border correctness. <!-- oc:id=item_am -->
1. The border class token must be explicitly passed into `InputGroupComposer`; missing that wiring previously caused a false sense that the ring implementation itself was broken. <!-- oc:id=item_an -->
1. Be careful with stacking/overflow. The shell needs decorative spill for glow/sparkle feel, but not in a way that visually enlarges layout. <!-- oc:id=item_ao -->
1. Focus handling is now split between CSS (`group-focus-within`) and local JS state in the sparkle overlay. Any future changes should keep those paths aligned. <!-- oc:id=item_ap -->
1. The user is highly visually sensitive here; final judgment is live rendered appearance, not just code cleanliness. <!-- oc:id=item_aq -->
