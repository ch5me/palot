# GenUI Artifact System <!-- oc:id=sec_aa -->

## TL;DR <!-- oc:id=sec_ab -->
> Turn one-off inline GenUI fences into first-class session artifacts with system-assigned IDs, pin/unpin affordances, editable props, and reusable placement surfaces.
>
> Deliverables:
> - Session-scoped artifact data model and persistence path
> - Artifact-aware GenUI rendering pipeline with generated IDs
> - Pin/unpin + list/update affordances for agent and user
> - New artifact surfaces in session widget and/or side-panel UI
> - Follow-up contract for agent references like `artifact_...`
>
> Estimated Effort: Large
> Parallel Execution: YES - 5 waves
> Critical Path: artifact model -> render capture -> UI surfaces -> agent affordances -> verification

---

## Context

### Original Request
Expand GenUI from inline fenced render-only output into reusable little widgets/artifacts. Audit how many components exist now, then plan toward a system where each generated GenUI element gets a stable ID, can be pinned or saved, can be referenced later by the agent, and can be incrementally edited instead of fully regenerated.

### Interview Summary
**Key Discussions**:
- Current inline GenUI and loading UX now feels good enough to use as a base.
- User wants every generated GenUI block to become addressable by ID.
- User wants pinning/saving into a persistent surface, potentially side panel or desktop-only inline-right area.
- User wants agent affordances for later reference: “pin that”, “save that”, “make that blue”.
- User wants planning now, not implementation in this turn.

**Research Findings**:
- Current GenUI registry has one real component today: `dag-sparkline` in `apps/desktop/src/renderer/genui/registry.ts` and `apps/desktop/src/renderer/genui/components/dag-sparkline.tsx`.
- Renderer pipeline is already centralized in `apps/desktop/src/renderer/genui/genui-renderer.tsx` and chat path already routes assistant text through it via `apps/desktop/src/renderer/components/chat/chat-turn.tsx`.
- Session widget placement already exists with `above-chat` and `chat-inline-right` zones in `apps/desktop/src/renderer/atoms/session-widgets.ts` and `apps/desktop/src/renderer/components/session-widgets/session-widget-shell.tsx`.
- Side panel tab architecture already exists in `apps/desktop/src/renderer/firefly-surface-registry.tsx` and `apps/desktop/src/renderer/components/side-panel/session-side-panel.tsx`.
- Pinned-item UX precedent exists in `apps/desktop/src/renderer/components/review/review-panel.tsx`.
- Durable archived architecture notes for tool-part-based artifact evolution exist in `.sisyphus/notepads/genui-artifact-archive.md`.

### Metis Review
**Identified Gaps** (addressed):
- Needed clearer split between session-scoped v1 and project/global persistence later.
- Needed explicit no-server-change first slice before bigger tooling ambitions.
- Needed separate treatment for user pinning vs agent-driven pinning/editing.
- Needed rollout path that works even while only one GenUI component exists today.

## Work Objectives <!-- oc:id=sec_ac -->

### Core Objective <!-- oc:id=sec_ad -->
Create a first-class GenUI artifact system that starts from today’s inline fence renderer but evolves it into stable, addressable, pinnable, and editable session artifacts.

### Concrete Deliverables <!-- oc:id=sec_ae -->
- Artifact domain model with IDs, placement metadata, source linkage, and mutable props
- Automatic artifact creation from resolved GenUI segments
- Session artifact registry/state layer
- User-facing artifact surfaces (widget and/or side panel)
- Agent-facing artifact reference contract and update/pin affordances
- Follow-up expansion plan for multiple component kinds beyond DAG

### Definition of Done <!-- oc:id=sec_af -->
- [x] Every resolved GenUI artifact can be assigned a system-generated ID and looked up in state
- [x] At least one session-visible pin surface can render pinned artifacts without losing chat ordering
- [x] Agent affordance contract exists for list/pin/unpin/update flows
- [x] Editing a prior artifact can update its props without requiring full manual regeneration semantics
- [x] Legacy inline rendering still works while artifact features roll out

### Must Have <!-- oc:id=sec_ag -->
- Session-safe IDs generated by system, not trusted from model output
- Strong linkage from artifact back to session/message/part origin
- Safe, typed payload validation per component kind
- Pin/unpin actions that are explicit in state and reversible
- Phased rollout that works with only `dag-sparkline` initially

### Must NOT Have (Guardrails) <!-- oc:id=sec_ah -->
- No arbitrary model-supplied React/HTML execution
- No dependence on immediate OpenCode server protocol changes for v1
- No artifact model that loses original source linkage
- No silent cross-session persistence assumptions without explicit scope choice
- No one-off “DAG special case” architecture that blocks future kinds
